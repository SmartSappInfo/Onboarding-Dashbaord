# Messaging Model Refactor Plan: Phase-by-Phase Implementation

## Overview
This plan refactors the messaging system to address architectural issues identified in the review: monolithic functions, tight coupling, complex variable resolution, poor error handling, scalability concerns, and security risks. With migration completed, we deprecate legacy code (e.g., dual-write logic, old variable names) and adopt the new schema fully.

**Goals:**
- Modular, testable, and maintainable code.
- Improved performance (parallelization, caching).
- Robust security (input validation, sanitization).
- Scalability for bulk messaging.
- Unified naming conventions (camelCase for variables).

**Assumptions:**
- Migration to new schema (entities, workspace_entities) is complete; legacy references (schools, prospects) are removed.
- Firestore remains the primary data store, but abstracted via repositories.
- No breaking changes to public APIs until Phase 5.

**Estimated Timeline:** 8-12 weeks, depending on team size and testing.

## Phase 1: Preparation and Planning (1-2 weeks)
**Objective:** Establish foundation without code changes; define interfaces and remove legacy code.

- **Tasks:**
  - Audit current codebase for legacy variables (e.g., `school_name` → `entity_name`, dual-write logic in `resolveContact`).
  - Define new interfaces: `MessageProvider` interface for delivery abstraction; `VariableResolver` for resolution logic.
  - Create repository interfaces: `ContactRepository`, `TemplateRepository`, `LogRepository`.
  - Remove deprecated code: Eliminate fallback to old schema (e.g., schools collection), enforce camelCase variables.
  - Set up monitoring: Add metrics for message sends, failures, and latency.

- **Deliverables:** Updated type definitions in `types.ts`, deprecation warnings in code comments.
- **Risks:** Breaking existing sends if legacy logic is removed prematurely.
- **Testing:** No functional changes; verify existing tests pass.

## Phase 2: Data Access Layer and Parallelization (2-3 weeks)
**Objective:** Abstract data access and optimize queries.

- **Tasks:**
  - Implement repositories: `ContactRepository` for entity/workspace data; `TemplateRepository` for templates/styles; `LogRepository` for audit logs.
  - Add caching: Use Redis/in-memory cache for org settings, global fields, and frequently accessed data.
  - Parallelize queries: Refactor `sendMessage` to use `Promise.all` for independent fetches (e.g., org snap + registrant snap).
  - Deprecate legacy: Remove hardcoded dual-entity logic; standardize variable names (e.g., rename `school_name` to `entity_name` everywhere).

- **Deliverables:** New `repositories/` folder with classes; updated `messaging-engine.ts` with parallel fetches.
- **Risks:** Cache invalidation issues; ensure atomicity in parallel ops.
- **Testing:** Unit tests for repositories; integration tests for parallel queries.

## Phase 3: Template Engine and Variable Resolution (2-3 weeks)
**Objective:** Enhance safety and consistency in variable handling.

- **Tasks:**
  - Upgrade template engine: Replace `resolveVariables` with Handlebars for advanced features (helpers, conditionals).
  - Implement variable validation: Schema-based validation for inputs; detect/resolve conflicts.
  - Simplify resolution: Define explicit precedence (e.g., input > entity > global); remove complex layering.
  - Sanitize outputs: Add HTML escaping for emails; prevent XSS in variable values.

- **Deliverables:** New `template-engine.ts` with Handlebars; updated `messaging-utils.ts`.
- **Risks:** Template syntax changes may break existing templates.
- **Testing:** Comprehensive tests for variable substitution, edge cases (e.g., missing vars, conflicts).

## Phase 4: Modularize Messaging Engine (2-3 weeks)
**Objective:** Break down monolith into composable functions.

- **Tasks:**
  - Split `sendMessage`: Into `resolveContext`, `renderContent`, `deliverMessage`, `auditMessage`.
  - Implement pipeline pattern: Use middleware for steps (suppression, tracking, scheduling).
  - Add error handling: Structured errors with context; circuit breakers for providers.
  - Deprecate legacy: Remove embedded business logic (e.g., hardcoded meeting tokens); move to configurable rules.

- **Deliverables:** Refactored `messaging-engine.ts`; new middleware classes.
- **Risks:** Refactoring may introduce bugs; ensure backward compatibility.
- **Testing:** Unit tests for each module; integration tests for full flow.

## Phase 5: Provider Abstraction and Scalability (2-3 weeks)
**Objective:** Decouple delivery and enable bulk sends.

- **Tasks:**
  - Abstract providers: Implement `MessageProvider` interface; wrap existing services (mNotify, Resend, etc.).
  - Add bulk support: Introduce queuing (Redis) for bulk messaging; batch deliveries where possible.
  - Optimize performance: Offload heavy ops (link tracking) to async jobs; add rate limiting.
  - Deprecate legacy: Remove direct imports; use provider registry.

- **Deliverables:** New `providers/` folder; updated delivery logic.
- **Risks:** Provider API changes may affect sends.
- **Testing:** Mock provider tests; load tests for bulk sends.

## Phase 6: Security, Validation, and Observability (1-2 weeks)
**Objective:** Harden security and improve monitoring.

- **Tasks:**
  - Input validation: Schemas for all inputs (templates, variables); reject malformed data.
  - Security enhancements: Sanitize all outputs; add CSP headers for emails.
  - Observability: Structured logging (levels, contexts); distributed tracing.
  - Deprecate legacy: Remove any remaining insecure practices (e.g., unescaped vars).

- **Deliverables:** Validation schemas; updated logging.
- **Risks:** Over-validation may block legitimate sends.
- **Testing:** Security audits; penetration tests for injection vulnerabilities.

## Phase 7: Cleanup, Deprecation, and Testing (1-2 weeks)
**Objective:** Finalize and validate.

- **Tasks:**
  - Remove all legacy code: Delete deprecated functions, variables, and comments.
  - Update documentation: Inline docs, API references.
  - Comprehensive testing: End-to-end tests; performance benchmarks.
  - Feature flags: Use flags for gradual rollout.

- **Deliverables:** Clean codebase; updated README/docs.
- **Risks:** Accidental removal of needed code.
- **Testing:** Full regression suite; user acceptance testing.

## Phase 8: Deployment and Monitoring (1 week)
**Objective:** Roll out and monitor.

- **Tasks:**
  - Staged deployment: Roll out to dev/staging; monitor metrics.
  - Post-deploy monitoring: Watch for errors, latency; rollback if needed.
  - User feedback: Collect input on new features (e.g., bulk sends).

- **Deliverables:** Production deployment; monitoring dashboard.
- **Risks:** Unforeseen issues in production.
- **Testing:** Production smoke tests.

## Dependencies and Team Considerations
- **Team:** 1-2 backend devs for core refactor; 1 QA for testing.
- **Tools:** Handlebars for templating; Redis for caching/queuing; Jest for testing.
- **Rollback Plan:** Feature flags for quick disable; git revert for code.
- **Success Metrics:** 50% reduction in message latency; 0 security incidents; all tests passing.
